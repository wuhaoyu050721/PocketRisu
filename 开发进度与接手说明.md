## 4. 关键环境与网络注意事项（坑点，务必遵守）

### 4.1 必须挂代理才能装/构建依赖
- 本机**直连 GitHub 会失败**（连接被重置 / 超时），原因是网络环境问题。
- **本地代理**：Clash.Verge 在 `127.0.0.1:7897`（HTTP 代理）。已验证可用。
- 凡是涉及下载 GitHub 资源的命令（`pnpm install`、`pnpm build`、`git clone/fetch`），**都要先设代理**：
  ```bash
  export HTTP_PROXY=http://127.0.0.1:7897
  export HTTPS_PROXY=http://127.0.0.1:7897
  ```
- 启动 `pnpm runserver` 本身不需要代理（纯本地），但带上也无害。

### 4.2 better-sqlite3 编译陷阱
- `better-sqlite3` 是原生 C++ 模块，优先从 GitHub 下预编译二进制包。
- **不挂代理** → 下载被重置 → 回退本地编译 → **本机没装 Visual Studio C++ Build Tools** → 编译失败 → `pnpm install` 整体报错退出。
- **解决**：挂代理重装，预编译包直接下载成功，绕开编译。当前依赖已装好，无需再动；只有 `pnpm install` 重新执行时才需注意。
- 若哪天代理也下不了，备选方案：装 VS Build Tools（"Desktop development with C++" 工作负载）后本地编译——但体积大，不推荐。

### 4.3 可忽略的警告（不影响运行）
- `The "pnpm" field in package.json is no longer read by pnpm` —— pnpm 版本迁移提示，无害。
- `Ignored build scripts: sharp@0.34.5` —— sharp 是可选图片处理库，小酒馆 核心不依赖，可忽略。需要时跑 `pnpm approve-builds`。
- 构建时 `Some chunks are larger than 2000 kB` —— bundle 体积警告，无害。

## 9. 局域网 / 手机访问（已配置成功 ✅）

场景：电脑插网线 + 手机连家里 Wi-Fi（同一台路由器），在手机上访问 小酒馆。

### 背景：IPv4 走不通，IPv6 走通
- **IPv4（`192.168.1.144:6001`）访问失败**：原因是路由器对"电脑有线 + 手机 Wi-Fi"这种组合做了 **AP 隔离**（无线设备无法访问有线设备）。这层隔离在路由器里，电脑端无法绕过（除非进路由器后台关 AP 隔离，或电脑改连 Wi-Fi）。
- **IPv6 直连成功** ✅：电脑和手机都是中国电信 `240e:` 段、同一路由器、同 IPv6 网段，可直连。

### 服务端无需改动
- `server.cjs` 用 `server.listen(port)` 不带 host，Node 默认双栈监听 `0.0.0.0` + `[::]`，IPv4/IPv6 都能接受连接。netstat 验证：`0.0.0.0:6001` 和 `[::]:6001` 都在 LISTENING。
- 已添加 Windows 防火墙入站规则 **`小酒馆 6001`**（TCP 6001，本地 IP 任何，覆盖域/专用/公用三种配置文件），IPv4 和 IPv6 入站都放行。规则持久，重启不丢。
- 本机自测：`http://127.0.0.1:6001`、`http://192.168.1.144:6001`、`http://[::1]:6001`、`http://[电脑IPv6]:6001` 全部返回 200。

### 手机访问方法（重点）

启动 `pnpm runserver` 后，**先查电脑当前的 IPv6 地址**（地址会变，每次访问前都要重新查）：
```bash
ipconfig | grep "IPv6" | grep "240e"   # 找 240e: 开头那行（电信公网 IPv6）
# 或更直接：
ipconfig
```
拿到形如 `240e:379:ada5:9600:1ad2:c89e:836c:272a` 的地址后，**手机浏览器输入**（注意方括号！）：
```
http://[240e:379:ada5:9600:1ad2:c89e:836c:272a]:6001
```

⚠️ **IPv6 地址在 URL 里必须用 `[ ]` 包住**，端口在右括号外，`http://` 不能省，否则手机当搜索词。这是最常见的"打不开"原因。

### 三个坑

1. **IPv6 地址会变**：家宽 IPv6 前缀（`240e:379:ada5:9600` 这段）运营商会定期更换，电脑重启/重新拨号也变。**每次访问前都要重新 `ipconfig` 查最新地址**。要固定需在路由器/系统层面配置静态 IPv6（较麻烦，一般不值得）。
2. **必须同网段同路由器**：手机和电脑都要是 `240e:` 段且连同一台路由器。若手机改用 4G/5G 移动数据，IPv6 段不同，直连不通。
3. **AP 隔离只挡 IPv4**：本案例 IPv4 被 AP 隔离挡住，但 IPv6 没被挡（不同隔离机制）。若哪天 IPv6 也不通，备选方案：
   - 进路由器后台（`http://192.168.1.1`）关 AP 隔离 → IPv4 `192.168.1.144:6001` 即可用
   - 或电脑拔网线改连同一 Wi-Fi → 重新 `pnpm runserver` → 用新 Wi-Fi IP 访问
   - 或用 小酒馆 自带的 cloudflared tunnel（代码里 `tunnelProcess` 支持生成公网临时域名）

### 11.2 国际化机制（务必先理解再改）

- **语言对象是合并出来的**，不是各语言一份完整副本：
  ```
  中文 = lodash.merge( 深拷贝(英文), cn.ts主翻译, cn.extra.ts补全 )
  ```
  - 英文（`en.ts`）是**键的完整来源**（base），保证任何翻译缺失都有兜底。
  - `cn.ts` 是上游原有的简体翻译（约 1342 行，**键不全**）。
  - `cn.extra.ts` 是本次新增的**补全层**，只放 `cn.ts` 缺失或仍为英文的键（约 460 条 UI + 304 条 help）。
  - 合并顺序：英文 → cn → extra，**后者覆盖前者**。所以不要在 `cn.extra.ts` 里重复 `cn.ts` 已正确翻译的键（无意义，且维护时易冲突）。
- **`help.*` 子树**：设置页每项的 ⓘ 帮助文案。`en.ts` 共 423 条，`cn.ts` 仅译 119 条，`extra` 补了 304 条。help 通过 `help.cn.ts` → `cn.ts` 的 `help: helpCn` 引入，extra 的 `help` 块会深合并到其上。
- **带参数的动态文案是函数**，如 `enterMessageToPersona: (name) => ...`。翻译时**必须保持函数签名一致**，只改字符串字面量，保留 `${}` 插值与逻辑（如 `backupSnapshotLimitsCurrent` 内有 if/else）。
- **技术术语保留英文**：Top P、JSON Schema、HypaMemory、Tokenizer、Seed、Bias、RisuRealm 等——与项目既有惯例一致，也是 42 个"与英文相同"键中刻意不译的部分。

### 11.3 辅助脚本（留在仓库，可复用）

放在 `scripts/` 下，基于 `sucrase` 编译 TS 在 Node 里求值（不依赖 vite）：

| 脚本 | 用途 |
|------|------|
| `scripts/compare-lang.cjs` | 对比 en / cn / zh-Hant 的叶子键，输出缺失、可被繁体覆盖、完全缺失三类，写 `scripts/lang-diff.json` |
| `scripts/verify-extra.cjs` | **改完 cn.extra.ts 后必跑**：验证 cn+extra 合并后，相对 en 还有多少键缺失/仍与英文相同。目标是两者都为 0 |

**改 en.ts 加了新键后**，跑 `node scripts/verify-extra.cjs`，它会列出需要在 `cn.extra.ts` 补译的键。

### 11.5 ⚠️ 旧数据库的语言字段（重要坑）

- 首次访问时 `save/risuai.db` 已建库，其 `language` 字段在**建库那一刻**就被写入（bootstrap 只在 `createdFreshDatabase` 时按浏览器语言设值，已建库则直接读 `data.language`）。
- 若本机旧库的 `language='en'`，刷新页面后界面仍会是英文——**这不是 bug**，是数据已固定。
- **解决**：进 设置 → 语言，把 UI 语言选成「中文」，关闭设置即生效。
- **全新安装**（无 `save/` 目录）则会自动默认中文。
- 若要让旧库也强制中文，可删 `save/risuai.db`（**会丢所有对话数据**，勿轻易做），或在 UI 里手动切。

### 11.6 硬编码英文字符串（长尾，未做穷尽）

- 本次只汉化了**最显眼**的几处（首页输入框、列表的 Back/Search 按钮、CharConfig 的 Model/API 占位符）。
- 项目里还有大量 Svelte 组件内**直接写死的英文**（`placeholder=`、`title=`、`aria-label=`、纯文本节点），这些不走 `language.xxx`，需要逐个排查替换。
- 排查命令：
  ```bash
  # 找硬编码英文 placeholder/title/aria-label
  grep -rnE 'placeholder="[A-Za-z][A-Za-z ]+"|title="[A-Za-z][A-Za-z ]+"|aria-label="[A-Za-z][A-Za-z ]+"' src/ --include=*.svelte
  ```
- 后续做移动端适配时顺手把这些清掉即可，不必单独开一轮。

### 11.7 后续汉化注意事项

1. **优先用 `language.xxx`，别在组件里写死中文**：否则切回英文/其他语言时会漏。
2. **新增 UI 文案时**：在 `en.ts` 加键 + 在 `cn.extra.ts` 加中文翻译（或直接加到 `cn.ts`），别只加中文不加英文键——英文是 base，缺了会 undefined。
3. **`en.ts` 更新后**：跑 `verify-extra.cjs` 看新增了哪些待译键。
4. **不要动繁体 `zh-Hant.ts`**：它是独立完整翻译，本次只参考其译法、未修改。
5. **merge 是浅层引用 `safeStructuredClone(languageEnglish)`**：每次 `changeLanguage` 都深拷贝英文 base，所以改 en.ts 重 build 即生效，无需重启服务（前端静态资源）。
6. **登录前界面**（密码框）：靠 `index.ts` 顶部默认导出 + `applyEarlyLanguage()` 生效，已设为中文。若改默认语言，同步改这两处 + bootstrap 兜底，三处一致。
